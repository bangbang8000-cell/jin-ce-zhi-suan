# Config Center Draft Action Design

## 1. Background

当前配置中心存在两个明显的使用问题：

- 一部分执行/测试类按钮会直接读取当前表单草稿，因此用户未保存也能立即生效
- 另一部分按钮仍依赖已保存配置或后端默认回读，导致用户必须先保存、关闭、重新进入配置中心，再点击按钮才会得到预期结果

这种行为不一致会直接带来误判：

- 用户以为按钮会使用自己刚改的值，实际却使用了旧配置
- 用户很难判断某个按钮到底是“按当前表单执行”还是“按已保存配置执行”

在 `工部增量同步` 场景中，这个问题尤为明显。用户希望在配置中心中直接修改参数、点击按钮执行，而不是先做一次保存流程。

同时，本轮还需要在配置中心的 `工部增量同步` 区域增加一个“手动更新股票池”按钮，用于触发股票列表刷新动作。

## 2. Goals And Non-Goals

### 2.1 Goals

- 统一配置中心内所有执行/测试类按钮的行为，改为严格使用当前未保存表单草稿
- 不再让执行类按钮依赖“先保存配置”作为前置条件
- 为执行类按钮增加前端必填字段校验，缺失时直接拦截并提示
- 在 `工部增量同步` 区域增加“更新股票池”按钮
- 让“更新股票池”和“增量同步”都遵循同一套草稿执行机制

### 2.2 Non-Goals

- 不改变“保存并生效”“重新加载”这类配置管理按钮的语义
- 不要求本次统一到所有弹窗、向导和主页面快捷按钮
- 不修改 `config.json` 结构
- 不把所有后端接口都改成禁止回读已保存配置
- 不在本次设计中实现股票池刷新脚本本身，脚本能力由独立 spec 负责

## 3. Scope

本次改动聚焦于：

- `dashboard.html` 中配置中心相关逻辑
- 配置中心内所有依赖配置执行的按钮
- `工部增量同步` 区域新增“更新股票池”按钮
- 对应最小后端接口透传补充

本次明确纳入统一的配置中心执行/测试类按钮清单：

- `testSelectedDataSourceConnectivity()`
- `testTushareConnectivity()`
- `testTdxConnectivity()`
- `testUnifiedLlmConnectivity()`
- `testStrategyManagerLlmConnectivity()`
- `testDataProviderLlmConnectivity()`
- `testWebhookDelivery()`
- `runHistorySync()`
- `runHistorySyncStockListRefresh()`
- `toggleHistorySyncScheduler()` 中的“启动定时同步”分支

本次明确排除但保持现有语义的按钮：

- `saveConfigCenter()`
- `loadConfigCenter()`
- `toggleHistorySyncScheduler()` 中的“停止定时同步”分支
- `停止同步`
- 非配置中心区域的执行按钮

本次不改动：

- 配置保存链路
- 配置加载链路
- 非配置中心入口按钮的统一行为
- `history_sync` 核心执行逻辑

## 4. Approaches

### 4.1 Recommended Approach: Strict Draft Action Mode With Lightweight Action Registry

推荐方案是为配置中心建立统一的“严格草稿执行模式”：

- 所有执行/测试类按钮都只读取 `collectConfigFromForm()` 的当前结果
- 前端按动作声明的必填字段做校验
- 若缺字段，则直接拦截，不发请求
- 若校验通过，则由前端把本次动作所需字段完整透传给后端
- 用一个轻量动作注册表统一管理每个动作的依赖、payload 构造和执行器，而不是继续在每个按钮函数中散落重复逻辑

优点：

- 行为一致，用户预期清晰
- 完全解决“先保存后生效”的体感问题
- 更适合配置调试类工作流

缺点：

- 需要梳理配置中心内各按钮的输入依赖
- 对执行型按钮的前端透传要求更高

### 4.2 Rejected Approach: Draft Preferred With Saved Fallback

让按钮优先使用草稿，但缺失时回退到已保存配置。

不采用原因：

- 用户仍无法直观看出哪些字段来自草稿，哪些字段来自旧配置
- 问题不会真正消失，只是从“总是旧值”变成“部分旧值”

### 4.3 Rejected Approach: Keep Backend Reload Model

继续让后端接口从 `ConfigLoader.reload()` 回读未显式传入的字段。

不采用原因：

- 无法保障配置中心“所见即所得”
- 用户依旧需要先保存配置才能稳定复现按钮行为

## 5. Detailed Design

### 5.1 Action Classification

本次统一对象不是配置中心中的全部按钮，而是“所有依赖配置执行动作的按钮”。

分为两类：

- 执行/测试类按钮
  - 例如：增量同步、更新股票池、启动定时同步、测试数据源连通性、三类模型连通性测试、Webhook 测试等
  - 这类按钮统一进入严格草稿执行模式

- 配置管理类按钮
  - 例如：保存并生效、重新加载、关闭弹窗、停止类按钮
  - 这类按钮不进入严格草稿执行模式，保留原语义

补充约束：

- “启动类动作”若依赖配置，必须纳入严格草稿执行模式
- “停止类动作”若仅用于终止正在运行的任务，不要求依赖草稿，不强行纳入必填校验

### 5.2 Unified Frontend Action Pipeline

在 `dashboard.html` 中抽象统一执行流程：

1. 调用 `collectConfigFromForm()` 获取当前草稿
2. 根据按钮动作类型读取对应的必填字段声明
3. 校验当前草稿是否满足必填要求
4. 若缺失字段，则在前端直接提示并中断
5. 若校验通过，则按当前草稿构造本次动作请求 payload
6. 发起请求并更新按钮状态与页面日志

这样每个按钮都不再各自零散地决定“是读草稿还是读保存值”，而是统一走同一套动作执行器。

建议实现为两个层次：

- 通用严格草稿入口
  - 负责读取草稿、校验必填项、打印统一日志、把 `config` 透传给后端
- 动作适配器
  - 负责每个按钮自己的 DOM 状态、payload 细节、成功/失败文案

这样既能保证行为统一，也不会把现有按钮的交互细节全部硬塞进一个超大函数。

### 5.3 Validation Strategy

采用严格模式：

- 只要按钮声明了必填字段，就必须在当前草稿中存在有效值
- 不允许回退到已保存配置
- 不把缺字段问题留给后端兜底

前端提示策略：

- 在按钮附近或状态文本中明确提示缺失字段
- 文案要直指字段名称，避免只提示“参数无效”

示例：

- `无法执行增量同步：缺少 history_sync.custom_start_time、history_sync.custom_end_time`
- `无法更新股票池：缺少股票池数据源选择`

### 5.4 Button Registry

建议为配置中心执行型按钮建立统一注册表，至少声明：

- `action_id`
- `button_id`
- `required_paths`
- `build_payload(draft)`
- `execute(payload, draft)`
- `before_log`
- `success_message`
- `failure_message`

这样可以避免后续继续出现“新增一个按钮就重新发明一套读取逻辑”的问题。

推荐把注册表控制在“轻量动作描述”层，不要求一次性抽象所有按钮动画和 DOM 细节。

建议首批注册项：

- `config_data_source_connectivity`
- `config_llm_evolution_connectivity`
- `config_llm_strategy_manager_connectivity`
- `config_llm_data_provider_connectivity`
- `config_webhook_test`
- `history_sync_run`
- `history_sync_stock_list_refresh`
- `history_sync_scheduler_start`

### 5.5 History Sync Area Enhancement

在配置中心 `工部增量同步` 区域中新增按钮：

- 文案：`更新股票池`

该按钮与现有：

- `增量同步`
- `停止同步`
- `定时同步开关`
- `刷新状态`

并列显示，但语义上归属于“准备同步输入数据”的前置动作。

执行规则：

- `更新股票池` 读取当前草稿中的股票池刷新相关字段
- 当前端校验通过后，触发股票池更新动作
- 更新成功后在页面日志和状态区明确提示最终使用的数据源与更新数量

同时补充 `history_sync` 区域的统一规则：

- `增量同步` 继续使用严格草稿执行模式
- `启动定时同步` 与 `增量同步` 使用同一份草稿依赖定义
- `停止定时同步`、`停止同步`、`刷新状态` 不要求强制走草稿校验

### 5.6 Backend Payload Strategy

后端接口为了兼容外部调用，仍可保留已有配置回读逻辑。

但配置中心入口应遵循以下原则：

- 前端尽量完整透传本次动作所需字段
- 后端不再被当作“从已保存配置补全当前草稿”的主要机制

换句话说：

- “草稿执行一致性”主要由前端保证
- “接口兼容性”继续由后端保留

本次对不同动作的 payload 约束如下：

- 数据源连通性测试：继续透传 `config`
- 模型连通性测试：前端应优先透传 `config`，若现有接口未接收，则在实现阶段补最小后端兼容
- Webhook 测试：前端应透传 `config`，避免仍只命中已保存 webhook 配置
- 增量同步 / 股票池更新 / 启动定时同步：继续沿用当前草稿透传模式

### 5.7 Status And Logging

统一动作执行后，前端应输出两类信息：

- 本次动作使用的是“当前草稿配置”
- 如果被前端拦截，明确打印缺失字段

例如：

- `使用当前草稿执行增量同步`
- `更新股票池已启动：provider=auto`
- `无法执行：缺少 history_sync.write_mode`

建议扩展到所有配置中心执行动作：

- `使用当前草稿执行数据源连通性测试`
- `使用当前草稿执行模型连通性测试`
- `使用当前草稿执行 Webhook 测试`
- `无法执行模型连通性测试：缺少 evolution.llm.base_url`

## 6. Data Flow

### 6.1 Execute/Test Action

1. 用户在配置中心修改字段但不保存
2. 点击执行/测试类按钮
3. 前端收集当前草稿
4. 前端校验必填字段
5. 校验通过则发起请求
6. 页面展示结果

对于“启动定时同步”一类动作，流程与普通测试按钮相同，只是最终请求会写入调度器配置并开启任务。

### 6.2 Config Management Action

1. 用户点击“保存并生效”或“重新加载”
2. 仍按现有配置管理流程执行
3. 不纳入“严格草稿执行模式”

## 7. Error Handling

- 若当前草稿无法收集，则按钮直接报错并停止执行
- 若必填字段缺失，则前端拦截，不发请求
- 若后端返回失败，则保留现有错误展示逻辑
- 若某个旧按钮尚未迁移到统一注册表，不允许静默沿用旧规则，应在实现阶段显式纳入或显式排除
- 若某个接口当前不支持 `config` 透传，应在实现阶段优先补后端最小兼容，而不是让前端回退到已保存配置

## 8. Testing Strategy

测试重点应覆盖：

- 按钮在未保存配置时仍可读取当前草稿并正确发起动作
- 必填字段缺失时，前端直接拦截
- 被拦截时不会向后端发请求
- `工部增量同步` 区域新增“更新股票池”按钮可见
- `增量同步` 与 `更新股票池` 都能使用当前草稿
- 数据源连通性测试使用当前草稿中的数据源配置
- 三类模型测试使用当前草稿中的模型配置
- Webhook 测试使用当前草稿中的 webhook 配置
- `启动定时同步` 使用当前草稿中的 `history_sync` 配置
- `停止同步`、`停止定时同步` 不因缺少草稿字段被拦截

若现有前端自动化测试基础不足，可以先做高价值静态验证与最小行为测试，但需要覆盖“草稿优先”和“缺失拦截”这两个核心行为。

## 9. Risks

- 若一次性迁移全部配置中心执行按钮，容易遗漏少量边缘动作，因此实现计划需要分批检查按钮清单
- 某些按钮的必填字段定义可能不明显，需要在实现阶段补充动作依赖清单
- 若按钮状态文案不清晰，用户可能误以为执行仍在使用已保存配置，因此需要明确提示“使用当前草稿执行”
- LLM 与 Webhook 测试若后端未接收 `config` 草稿透传，前端改造后仍会表现为“像是读取旧配置”，这需要在实现阶段优先补兼容

## 10. Acceptance Criteria

满足以下条件则视为完成：

- 配置中心内所有执行/测试类按钮统一使用当前草稿执行
- 不再要求用户先保存后退出再重新进入配置中心才能生效
- 缺失必填字段时前端直接拦截，并提示缺失项
- `工部增量同步` 区域新增“更新股票池”按钮
- `保存并生效` 与 `重新加载` 继续保持原有配置管理语义
- `测试数据源连通性`、三类 `测试模型`、`Webhook 测试`、`增量同步`、`更新股票池`、`启动定时同步` 均纳入统一严格草稿执行链路
- `停止同步`、`停止定时同步`、`刷新状态` 保持原有语义，不因草稿缺失而失效
